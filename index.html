Field guide · zero to fluent

You already know 90% of PostgreSQL, from MySQL

SQL is SQL. SELECT, JOIN, GROUP BY, indexes, transactions — you carry all of that across untouched. This guide skips the basics and teaches the other 10%: the dialect quirks, the gotchas that'll trip you on day one, and the genuinely better tools waiting on the other side. Think of it as a phrasebook for a language you almost already speak.

01

The mental model shift

Before any syntax — what's fundamentally different under the hood.

Most MySQL→Postgres friction isn't syntax — it's expectations. A handful of architectural facts explain almost every surprise you'll meet later, so internalise these first.

  • No storage engines. There's no InnoDB vs MyISAM decision. Postgres has one engine, and it always does MVCC, real transactions, and foreign keys. The "which engine?" question simply disappears.
  • It's stricter and more standards-compliant. Postgres tends to reject ambiguous things MySQL silently accepts — loose GROUP BY, implicit type coercions, invalid dates like '0000-00-00'. This feels annoying at first and saves you from real bugs later.
  • DDL is transactional. You can wrap CREATE TABLE, ALTER TABLE, even DROP inside BEGIN … ROLLBACK and undo them. In MySQL, DDL forces an implicit commit. This single difference changes how you write migrations.
  • Extensibility is the personality. Custom types, operators, index methods, procedural languages, and extensions (PostGIS, pg_trgm, pgvector…) are first-class. Postgres is less a database you configure and more a platform you extend.
  • One connection = one database. Unlike MySQL where USE otherdb hops freely, a Postgres connection is bound to a single database. Cross-database queries aren't a thing (you use schemas within one database instead — module 03).
★
Reframe, don't relearnEvery SELECT you've ever written still works. You're not a beginner here — you're an expert learning a precise dialect. Read the rest of this guide as "diffs," not "docs."
02

psql & the toolbelt

Your muscle memory for the mysql CLI needs a small retraining.

The interactive client is psql. Instead of SQL statements like SHOW, Postgres uses backslash meta-commands — terse, fast, and worth memorising.

MySQL CLI→
mysql -u root -p mydb

SHOW DATABASES;
USE shop;
SHOW TABLES;
DESCRIBE orders;
SHOW CREATE TABLE orders;
SHOW INDEX FROM orders;
psql
psql -U postgres -d mydb     -- or: psql "postgresql://user:pass@host:5432/mydb"

\l            -- list databases
\c shop       -- connect (re-connects!)
\dt           -- list tables
\d orders     -- describe table + indexes
\d+ orders    -- describe, verbose
\di           -- list indexes

The meta-commands you'll actually use

  • \? list all meta-commands · \h CREATE TABLE SQL help for a statement
  • \dn schemas · \du roles/users · \df functions · \dv views
  • \x toggle expanded display (lifesaver for wide rows — like MySQL's \G)
  • \timing show query time · \i file.sql run a script · \e edit in $EDITOR
  • \copy table TO 'out.csv' CSV HEADER — client-side bulk import/export
ⓘ
Dump & restoremysqldump → pg_dump / pg_restore. For a quick logical backup: pg_dump -Fc mydb > mydb.dump, restore with pg_restore -d newdb mydb.dump.
03

Databases, schemas & search_path

The concept with no clean MySQL equivalent. Get this right early.

In MySQL, "database" and "schema" are synonyms and you switch between them freely. In Postgres the hierarchy has an extra layer: a cluster holds many databases, and each database holds many schemas (namespaces), which hold your tables.

  • A new table goes into the public schema by default.
  • The search_path decides which schemas are checked, in order, when you write an unqualified table name — like a PATH variable for tables.
  • You cannot join across databases in one query. If you were using multiple MySQL databases to namespace one app, use Postgres schemas instead — they live in the same database and join freely.
PostgreSQL
CREATE SCHEMA reporting;
CREATE TABLE reporting.daily_sales (id int, total numeric);

-- unqualified names resolve via search_path:
SHOW search_path;                      -- "$user", public
SET search_path TO reporting, public;  -- check reporting first now

-- cross-schema joins are normal (same database):
SELECT * FROM public.orders o
JOIN reporting.daily_sales d ON d.id = o.id;
⚠
"relation does not exist" after creating itNine times out of ten your table is in a schema that isn't on the search_path. Either schema-qualify the name (reporting.daily_sales) or add the schema to your path.
04

Quoting & case sensitivity

The #1 source of "but it worked in MySQL!" bug reports.

Two quoting rules that differ from MySQL and cause real pain if you don't know them up front.

MySQL→
-- backticks quote identifiers
SELECT `userName` FROM `Users`;

-- double quotes often = strings (in default mode)
SELECT "hello world";   -- a string

-- string LIKE is case-INsensitive by default
SELECT * FROM users
WHERE name LIKE 'alice';   -- matches 'Alice'
PostgreSQL
-- DOUBLE quotes quote identifiers; SINGLE quotes = strings
SELECT "userName" FROM "Users";   -- exact case preserved
SELECT 'hello world';             -- a string

-- unquoted identifiers fold to LOWERCASE:
CREATE TABLE Users (userName text);  -- becomes users(username)
SELECT userName FROM Users;          -- works (folded to lowercase)

-- LIKE is case-SENSITIVE; use ILIKE for insensitive:
SELECT * FROM users WHERE name ILIKE 'alice';  -- matches 'Alice'
⚠
The double-quote trapIf you ever create a table with CREATE TABLE "Users" (quoted, mixed case), you must always quote it with exact case forever: SELECT * FROM "Users". Mismatched case → "relation does not exist". The fix is simple: never quote identifiers, and use snake_case everywhere. Postgres folds unquoted names to lowercase, so snake_case sidesteps the whole problem.
05

Data types

Mostly familiar, with a few that don't exist and a few that are upgrades.

Numbers, dates and strings map across cleanly. The surprises are the things MySQL has that Postgres doesn't, and the things Postgres does much better.

  • No UNSIGNED. Pick a wider type (use bigint) or add a CHECK (x >= 0) constraint.
  • No TINYINT / MEDIUMINT. Use smallint, integer, bigint. For 0/1 flags use a real boolean.
  • boolean is a true type, not an alias for TINYINT(1). Accepts TRUE/FALSE, 't'/'f', 1/0.
  • text has zero penalty. varchar(n) is just text with a length CHECK. Default to text unless you truly need a length limit. There is no performance difference.
  • Real ENUM types. An enum is a standalone, reusable type — not an inline column definition.
  • timestamptz stores an instant in UTC and converts to the session timezone. Prefer it over plain timestamp for anything time-zone aware. MySQL's DATETIME ≈ timestamp; its TIMESTAMP ≈ timestamptz.
  • Native uuid, json/jsonb, arrays, ranges, inet/cidr — types MySQL either lacks or fakes with strings.
MySQL→
CREATE TABLE account (
  id        INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  is_active TINYINT(1) DEFAULT 1,
  role      ENUM('admin','user') NOT NULL,
  bio       VARCHAR(2000),
  created   DATETIME DEFAULT CURRENT_TIMESTAMP
);
PostgreSQL
CREATE TYPE user_role AS ENUM ('admin','user');  -- reusable type

CREATE TABLE account (
  id        bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  is_active boolean DEFAULT true,
  role      user_role NOT NULL,
  bio       text,                       -- no length limit needed
  created   timestamptz DEFAULT now()
);
06

Auto-increment & sequences

Where did AUTO_INCREMENT and LAST_INSERT_ID() go?

Postgres has no AUTO_INCREMENT keyword. Auto-numbering is powered by sequences — independent objects that hand out numbers. Two ways to wire one up:

  • GENERATED ALWAYS AS IDENTITY — the modern, SQL-standard way. Prefer this.
  • serial / bigserial — the classic shorthand that auto-creates a sequence. Still everywhere in older code.
MySQL→
CREATE TABLE post (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(200)
);

INSERT INTO post (title) VALUES ('Hello');
SELECT LAST_INSERT_ID();   -- get the new id
PostgreSQL
CREATE TABLE post (
  id    bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  title text
);

-- RETURNING gives you the new row's values in one round-trip:
INSERT INTO post (title) VALUES ('Hello')
RETURNING id;              -- way better than LAST_INSERT_ID()
★
RETURNING beats LAST_INSERT_ID()It works on multi-row inserts, on UPDATE and DELETE too, and returns any columns — not just the last id. No second query, no race conditions. See module 08.
07

Syntax that bites

The everyday statements where the dialect diverges.

These are the small differences you'll hit within your first hour of writing real queries.

MySQL→
-- pagination
SELECT * FROM t LIMIT 20, 10;    -- offset 20, take 10

-- string concat
SELECT CONCAT(first, ' ', last);

-- null fallback
SELECT IFNULL(nickname, 'n/a');

-- date formatting
SELECT DATE_FORMAT(d, '%Y-%m');

-- regex
SELECT * FROM t WHERE name REGEXP '^A';

-- current time
SELECT NOW();
PostgreSQL
-- pagination (no "20,10" form)
SELECT * FROM t LIMIT 10 OFFSET 20;

-- string concat: || operator (CONCAT also works)
SELECT first || ' ' || last;

-- null fallback: standard COALESCE
SELECT COALESCE(nickname, 'n/a');

-- date formatting: TO_CHAR
SELECT TO_CHAR(d, 'YYYY-MM');

-- regex: ~ (case-sensitive), ~* (insensitive)
SELECT * FROM t WHERE name ~ '^A';

-- current time
SELECT now();
⚠
GROUP BY is strictPostgres requires every selected non-aggregated column to appear in GROUP BY (the SQL standard). MySQL historically let you omit them. If you relied on loose grouping, you'll now get an error — list the columns, or wrap them in an aggregate. (Modern MySQL with only_full_group_by behaves the same way.)
ⓘ
Also worth knowingBoolean columns compare with WHERE is_active (no = 1 needed). DISTINCT ON (col) grabs the first row per group — a Postgres-only shortcut. Both engines now support CTEs and window functions identically.
08

Upserts & RETURNING

Translating ON DUPLICATE KEY UPDATE and REPLACE INTO.

Postgres spells "insert, or update if it already exists" as INSERT … ON CONFLICT. It's more explicit than MySQL's version — you name the conflicting column(s), and reference the would-be-inserted row via the special EXCLUDED table.

MySQL→
INSERT INTO inventory (sku, qty)
VALUES ('A1', 5)
ON DUPLICATE KEY UPDATE
  qty = qty + VALUES(qty);

-- REPLACE: delete + re-insert
REPLACE INTO inventory (sku, qty)
VALUES ('A1', 5);
PostgreSQL
INSERT INTO inventory (sku, qty)
VALUES ('A1', 5)
ON CONFLICT (sku) DO UPDATE
  SET qty = inventory.qty + EXCLUDED.qty
RETURNING sku, qty;        -- see the result, optionally

-- "insert, ignore if exists":
INSERT INTO inventory (sku, qty)
VALUES ('A1', 5)
ON CONFLICT (sku) DO NOTHING;
ⓘ
EXCLUDED = the rejected rowInside DO UPDATE, EXCLUDED.qty is the value you tried to insert, and the bare column / tablename.qty is the existing row. There's no direct REPLACE INTO equivalent — and you rarely want one, since ON CONFLICT updates in place instead of delete-then-insert (which would reset sequences and fire triggers).
09

JSONB & arrays

One of the biggest reasons teams move to Postgres.

MySQL has a JSON type, but Postgres' jsonb is a different league: binary-stored, indexable with GIN, and packed with operators and containment querying. There are also true arrays as a column type.

PostgreSQL · JSONB
CREATE TABLE event (id bigint GENERATED ALWAYS AS IDENTITY, data jsonb);

INSERT INTO event (data) VALUES
  ('{"type":"click","user":{"id":7,"plan":"pro"},"tags":["a","b"]}');

-- ->  returns jsonb   |   ->> returns text
SELECT data->'user'->>'plan'        AS plan,     -- 'pro'
       data->'user'->>'id'          AS user_id,  -- '7'
       data #>> '{user,id}'         AS deep       -- path access
FROM event;

-- @> containment: "does data contain this shape?" (GIN-indexable)
SELECT * FROM event WHERE data @> '{"type":"click"}';

-- ? key-exists; build/modify with jsonb_set, jsonb_build_object
SELECT * FROM event WHERE data->'user' ? 'plan';

-- index every key/value for fast containment queries:
CREATE INDEX idx_event_data ON event USING GIN (data);
PostgreSQL · native arrays
CREATE TABLE article (id int, tags text[]);
INSERT INTO article VALUES (1, ARRAY['sql','postgres','db']);

SELECT * FROM article WHERE 'postgres' = ANY(tags);  -- membership
SELECT * FROM article WHERE tags @> ARRAY['sql'];     -- contains
SELECT id, unnest(tags) AS tag FROM article;          -- explode to rows
★
Default to jsonb, not jsonUse jsonb unless you specifically need to preserve key order and whitespace exactly. Only jsonb supports indexing and the rich operators above.
10

Indexes & full-text search

More index types, no FULLTEXT keyword.

B-tree is still the default and covers most needs. Beyond it, Postgres offers specialised index methods you reach for deliberately: GIN (jsonb, arrays, full-text), GiST (geometry, ranges), BRIN (huge append-only tables), and Hash.

  • Partial indexes — index only the rows you query: CREATE INDEX … WHERE archived = false;
  • Expression indexes — index a computed value: CREATE INDEX … ON users (lower(email));
  • CREATE INDEX CONCURRENTLY — build without locking writes (essential in production).
MySQL→
-- full-text in MySQL
ALTER TABLE doc
  ADD FULLTEXT(body);

SELECT * FROM doc
WHERE MATCH(body)
AGAINST('postgres tips');
PostgreSQL
-- full-text via tsvector / tsquery + GIN
CREATE INDEX idx_doc_fts ON doc
  USING GIN (to_tsvector('english', body));

SELECT * FROM doc
WHERE to_tsvector('english', body)
   @@ websearch_to_tsquery('english', 'postgres tips');
ⓘ
Fuzzy / typo searchFor "LIKE %term%" or typo-tolerant matching, enable the pg_trgm extension and add a GIN trigram index — it makes leading-wildcard ILIKE queries fast, which MySQL can't index at all.

Functions & PL/pgSQL

Stored logic, dollar-quoting, and the DO block.

Postgres leans on functions (which return values and can be used in queries) more than on procedures. The default procedural language is PL/pgSQL. The signature quirk is dollar-quoting ($$ … $$), which frees you from escaping quotes inside the body and removes the need for MySQL's DELIMITER dance.

MySQL→
DELIMITER //
CREATE FUNCTION add_tax(p DECIMAL(10,2))
RETURNS DECIMAL(10,2)
DETERMINISTIC
BEGIN
  RETURN p * 1.2;
END //
DELIMITER ;
PostgreSQL
CREATE FUNCTION add_tax(p numeric)
RETURNS numeric
LANGUAGE plpgsql AS $$
BEGIN
  RETURN p * 1.2;
END;
$$;

-- call it like any built-in:
SELECT add_tax(100);   -- 120.0

-- one-off anonymous block (no function needed):
DO $$ BEGIN RAISE NOTICE 'hi from a DO block'; END $$;
ⓘ
Triggers split into two piecesYou write a trigger function returning trigger, then attach it with CREATE TRIGGER … EXECUTE FUNCTION fn(). Inside, NEW and OLD work just like MySQL.
12

Transactions, MVCC & VACUUM

Two superpowers and one chore you must understand.
  • Transactional DDL (superpower). Schema changes can live inside a transaction and be rolled back. Migrations that fail halfway leave nothing behind.
  • MVCC always-on (superpower). Readers never block writers and writers never block readers. An UPDATE writes a new row version rather than locking the old one.
  • VACUUM (the chore). Because updates/deletes leave "dead" row versions, Postgres reclaims them with VACUUM. autovacuum handles this automatically in the background — you mostly just need to know it exists and not disable it.
PostgreSQL
BEGIN;
  ALTER TABLE orders ADD COLUMN refunded boolean DEFAULT false;
  UPDATE orders SET refunded = true WHERE status = 'returned';
  -- changed your mind? this undoes the ALTER too:
ROLLBACK;     -- or COMMIT;

-- inspect a query plan with real timings:
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders WHERE customer_id = 42;

-- refresh planner statistics after big data changes:
ANALYZE orders;
★
Row locking when you need itSELECT … FOR UPDATE locks selected rows for the rest of the transaction (job queues, balance updates). Add SKIP LOCKED to build a contention-free worker queue — a pattern that's clean and reliable in Postgres.
